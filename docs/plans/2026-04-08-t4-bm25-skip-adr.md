# ADR: Skip T4 BM25 — empirical gate failure (2026-04-08)

**Status:** Decided. T4 BM25 is not implemented. The current Jaccard-style scorer
in `src/engine/solution-matcher.ts:calculateRelevance` remains the production
ranker.

**Reference plan:** `docs/plans/2026-04-07-synonym-map-round3-validation.md` —
Task 4 ("Add BM25 Only If Step 1-3 Metrics Plateau"), Step 4 explicitly states:
*"If BM25 does not beat Step 3 materially, keep the simpler scorer."*

**Reference baseline:** `ROUND3_BASELINE` in `src/engine/solution-matcher.ts`,
fixture v2 in `tests/fixtures/solution-match-bootstrap.json` (53+16+14 queries).

## Context

After T1 (bootstrap eval), T2 (term-normalizer migration), and T3 (privacy-
preserving ranking log), the v1 fixture (41+10+10) showed every metric at
ceiling: `recallAt5 = 1.0`, `mrrAt5 = 1.0`, `noResultRate = 0.0`,
`negativeAnyResultRate = 0.1`. This made the T4 plateau-gate question unanswerable
— "is this a plateau or is the fixture too easy?" had no data behind it.

T3.5 expanded the fixture to v2 with 22 deliberately hard cases:
- 12 hard positive queries (8 strong + 4 multi-canonical / compound-tag tug-of-
  war queries that rank @2-3)
- 6 Korean subtle paraphrases
- 4 tricky negative queries (dev-adjacent words: `performance`, `system`,
  `database`, `validation`)

The expanded fixture revealed:
- `recallAt5`: 1.0 → 1.0 (still ceiling — recall is genuinely solved)
- `mrrAt5`: 1.0 → 0.969 (small ranking room)
- `noResultRate`: 0.0 → 0.0
- `negativeAnyResultRate`: 0.1 → 0.357 (3.5× jump — the BIG signal)

The negativeAnyResultRate jump on the v2 fixture initially looked like an
unambiguous T4 BM25 + IDF target. The hypothesis: BM25 IDF would down-weight the
common dev-adjacent words ('performance', 'database', 'system') that drive both
the ranking failures and the false positives.

## Decision

Skip T4 BM25. **The hypothesis was wrong.** Empirical prototypes of BM25 (and two
hybrid variants) all underperformed or matched the current scorer on every metric:

| Approach | positive mrr | paraphrase mrr | negativeAnyResultRate | Notes |
|---|---|---|---|---|
| **Current Jaccard (baseline)** | **0.959** | **1.000** | **0.357** | reference |
| Naive BM25 (k1=1.2, b=0.75, IDF over raw tags) | 0.949 ⬇ | 0.969 ⬇ | 0.357 = | + new failure on "preventing sql injection" |
| Hybrid (Jaccard × clamped IDF [0.5–1.5]) | 0.949 ⬇ | 1.000 = | 0.357 = | IDF noise dominates |
| Precision filter (≥2 distinct families) | 0.868 ⬇⬇ | 0.875 ⬇⬇ | 0.071 ⬆ | recall collapse: 7 positives + 2 paraphrases zero-out |
| Soft penalty (× 0.3 when low dev-spec query × low match) | 0.959 = | 0.969 ⬇ | 0.357 = | DEV_VOCAB partial-match too lenient |

None of the variants improve **any** metric without regressing another. The
plan's gate ("BM25 must improve at least one of mrrAt5 / top-1 precision /
no-result rate without increasing false positives") is not met.

## Root cause

BM25 is the wrong tool for the **structural** properties of this matcher:

1. **Corpus is too small for IDF to be informative.** The starter corpus has
   N=15 solutions. Lucene-style IDF `ln(1 + (N - n + 0.5) / (n + 0.5))` has its
   useful range when terms appear in 30-70% of documents. With N=15, almost
   every tag appears in only 1-2 solutions, so IDF clusters in the 1.6-2.4 range
   — too noisy to differentiate meaningful from coincidental matches. The
   hybrid variants with clamped IDF `[0.5, 1.5]` confirm this: even after
   clamping, the IDF signal is dominated by tag rarity noise.

2. **Tag-based matching has TF ∈ {0, 1} by construction.** BM25's TF saturation
   `(f × (k1+1)) / (f + k1 × lenNorm)` collapses for binary TF. The remaining
   signal is `IDF × lenNorm`, which is essentially "weighted tag count, length-
   normalized" — the same structure as the current scorer with `tagWeight()`
   and Jaccard denominator. There is no algorithmic distinction left for BM25
   to exploit.

3. **The false positives are semantic, not statistical.** "performance review
   meeting notes" matches caching-strategy because `performance` IS a legitimate
   dev tag AND a legitimate English noun. No frequency-based weighting can
   distinguish "user typed dev term in dev context" from "user typed the same
   word in a non-dev context". The fix has to come from query-side semantic
   classification, not ranker-side TF-IDF math.

4. **The mrrAt5 ranking failures are compound-tag tokenization artifacts, not
   ranking-formula failures.** The 4 hard positives all rank @2-3 because:
   - `extractTags` strips hyphens before lookup, so query `api-key` becomes
     `[api, key]` and never reaches the solution's `api-key` tag.
   - `partial substring matching` pulls `code` into `code-review` regardless
     of relevance.
   - Single-tag generic matches (`function`, `refactor`) outscore specific
     compound tags (`red-green-refactor`, `관심사분리`) due to the Jaccard
     denominator favoring smaller unions.

   None of these are TF-IDF problems. BM25 IDF weights individual terms; it
   has no concept of compound vs simple tags or hyphen-stripped tokenization.

## What would actually help (Round 4 candidates, NOT this PR)

These are the real T4-equivalent improvements that the data points to. None
are in scope for the Round 3 plan; they become Round 4 candidates, prioritized
by the same metric-driven gate this ADR uses.

1. **Tokenizer fix for compound tags** — change `extractTags` (currently at
   `src/engine/solution-format.ts:283`, regex `[^가-힣a-z0-9\s]`) to keep
   hyphenated identifiers as single tokens (`api-key` stays atomic), or to
   index the corpus with both the hyphen-stripped and the atomic forms. This
   fixes the 4 hard positive ranking failures directly and is a well-scoped,
   surgical change. Estimated impact: positive `mrrAt5` 0.959 → ~0.99.
   Actionable: yes.

2. **Phrase/n-gram matcher overlay** — recognize 2-word phrases like
   `performance review`, `system architecture`, `database backup` as
   non-dev-context signals and demote matches when they appear in the query.
   Minimum spec: hand-curate the phrase list from (a) the 5 known-trigger
   negatives in fixture v2 ("performance review", "system architecture",
   "database backup", "validation … insurance", "solar system") plus (b) a
   manual review of the top-50 corpus tags for English homographs. Phrase
   normalization: lowercase + NFC, English-only as a starting point (Korean
   added later if needed). No ML, no external data sources, ~50 phrases at
   start. Estimated impact: `negativeAnyResultRate` 0.357 → ~0.10.
   Actionable: yes (with the spec above).

3. **Query-side dev specificity classifier** — a precomputed list of "high
   signal dev terms" vs "ambiguous English-and-dev terms"; require at least
   one high-signal term in the query before any match is returned.
   Conceptually similar to the precision filter prototype that ran during
   this investigation, but with a curated vocabulary instead of automatic
   family counting (the auto-family approach killed recall in the prototype).
   Actionable: yes, but requires the vocabulary curation work upfront.

(For "wait until corpus grows to N≥100" — this is a passive trigger, not a
candidate. It's captured in the Reversal triggers section below, not here.)

## Reversal triggers

This decision should be re-evaluated automatically when ANY of the following
become true. The goal is to prevent "skip once → skip forever" calcification.

1. **Production corpus grows to N ≥ 100 solutions on a representative user.**
   BM25 IDF becomes statistically informative at this corpus size. The
   `negativeAnyResultRate` jump from "1 of 15 solutions has tag X" to "12 of
   100 solutions have tag X" changes IDF from ~2.4 to ~1.1, which is the
   range where BM25 actually does work as designed. Re-prototype against
   the production corpus, not the starter pack, before deciding.

2. **A compound-tag tokenizer fix lands first.** If Round 4 candidate #1
   ships and removes the 4 hard-positive ranking failures, the residual
   `mrrAt5` and `negativeAnyResultRate` weakness changes shape — re-test
   BM25 on the new residual rather than the current one.

3. **`negativeAnyResultRate` regresses past 0.5 in the bootstrap eval** for
   any reason (new fixture additions, term-normalizer expansion, etc.).
   This signals the false-positive problem has structurally changed and
   the BM25 question must be re-asked even if the original 4-variant data
   still holds for the previous shape.

4. **`recallAt5` drops below 0.95 on the v2 fixture without a clear
   explanation.** Currently recall is at ceiling and locked. If a future
   change drops it, BM25's IDF-based ranking might recover ground that
   tag intersection cannot.

These triggers are intentionally narrow. Routine fixture edits, baseline
updates, or unrelated matcher tweaks do NOT warrant re-litigating this
decision — only a structural shift in one of the four signals above does.
