## 소개

고객에게 더 유용한 방향을 먼저 고민하고, 사람들과 함께 어려운 문제를 설계 중심으로 풀어가는 프론트엔드 개발자입니다. 저는 구현 속도 자체보다 문제를 어떻게 정의하고, 어떤 구조로 풀어야 제품과 팀 모두에 오래 남는지를 더 중요하게 생각합니다. 그래서 자연스럽게 공통 라이브러리, 시각화, 점진적 전환 구조, 팀 자산화 같은 문제를 많이 맡아왔습니다.

실무에서 반복해서 해온 일은 크게 세 가지였습니다.

- 복잡한 데이터와 구조를 더 빠르게 이해할 수 있는 화면으로 바꾸는 일
- 한 번 푼 문제를 공통 라이브러리나 시스템으로 정리해 반복 비용을 줄이는 일
- 조직이 무리 없이 새로운 기술로 옮겨갈 수 있도록 전환 경로를 만드는 일

저는 혼자 빠르게 답을 내기보다, 팀과 함께 문제를 해석하고 더 나은 구조를 만드는 과정을 좋아합니다. 좋은 해결책은 개인의 경험으로 끝나는 것이 아니라 팀의 공통 자산이 되어야 한다고 생각하고, 실제로 라이브러리, 가이드, 하네스, 공통 구조를 만드는 방식으로 그 관심사를 실무에 연결해왔습니다.

## (주) 인티그레이션 | Frontend Developer | 2024.03 ~ Present

인티그레이션에서는 시각화 중심 제품 경험을 더 유연하고 일관되게 만들기 위한 공통 기반을 만드는 데 집중했습니다. 동시에 Vue 3 중심 구조에서 React로 넘어갈 수 있는 전환 경로를 만들고, 좋은 방법이 개인 경험으로만 남지 않도록 팀 자산화 흐름까지 함께 고민했습니다. 이 시기의 핵심 키워드는 `공통화`, `전환 전략`, `팀 자산화`였습니다.

### 시각화 라이브러리 (Intiviz)

**왜 이 프로젝트가 필요했나요?**

- 스쿼드별로 ECharts, Chart.js 등 차트 구현 방식이 제각각이라 제품별 스타일 일관성과 확장성이 떨어졌습니다.
- 커스텀 차트 요구가 생길 때마다 화면 단위로 다시 구현해야 했고, 기존 라이브러리만으로는 디자인 요구사항을 충분히 따라가기 어려웠습니다.
- 결국 시각화를 기능 단위 구현이 아니라 공통 시스템으로 다뤄야 한다고 판단했습니다.

**저는 이렇게 풀었습니다.**

- D3.js와 TypeScript 기반의 framework-agnostic 시각화 라이브러리를 설계했습니다.
- 특정 프레임워크에 종속되지 않는 순수 TypeScript 구조를 선택했고, Atomic Design에서 영감을 받아 시각화 요소를 atom 단위 함수로 분리했습니다.
- 이때 핵심은 유연성과 사용성을 동시에 잡는 것이었습니다. API를 Atom과 Molecule의 2-Layer 구조로 나눠 저수준 요소 조합과 완성형 차트 사용을 모두 지원하도록 만들었고, 실제로 16개 draw primitive와 4개 chart component 규모로 구성했습니다.
- React/Vue 공용으로 사용할 수 있는 Headless 구조를 설계했고, Bar, Line, Area, Donut, Radar를 포함해 8종 차트와 Composite Chart까지 확장했습니다.
- 옵션 증가로 생긴 Props Explosion은 xAxis, yAxis, hover 같은 도메인별 config 구조와 TypeScript utility type으로 정리했습니다.

**무엇이 달라졌나요?**

- 현재 3~4개 제품에서 공통으로 사용하는 시각화 기반이 생겼습니다.
- 제품마다 새로운 차트를 처음부터 다시 만드는 대신, 공통 기반 위에서 필요한 표현을 primitive와 config 조합으로 확장하는 흐름으로 바꿀 수 있었습니다.
- 이 경험은 저에게 차트를 몇 종 만들었다는 의미보다, 시각화를 제품 기능이 아니라 공통 시스템으로 설계하고 운영해본 경험이라는 점에서 더 중요했습니다.

### Module Federation 기반 마이크로 프론트엔드

**왜 이 프로젝트가 필요했나요?**

- Vue 3 중심 구조로 인한 채용·유지보수 부담이 커지고 있었지만, 전체를 한 번에 React로 옮기기에는 비용과 위험이 너무 컸습니다.
- 보안 제약 때문에 iframe을 대안으로 쓰기도 어려웠습니다.
- 모놀리식 레포가 커지며 빌드 시간이 늘고 배포 영향 범위도 넓어져, 기능 단위로 독립 배포 가능한 구조가 필요했습니다.

**저는 이렇게 풀었습니다.**

- Vite + Federation 기반으로 Vue 3 host에서 BI 기능을 React remote app 2개로 분리하는 구조를 설계했습니다.
- 목표는 단순히 React 화면을 띄우는 것이 아니라, 기존 서비스를 깨지 않으면서 React가 독립적으로 배포되고 공존할 수 있는 경로를 만드는 것이었습니다.
- 실제로 어려웠던 지점은 host/remote 경계에서 어느 수준까지 런타임과 상태를 분리할지 정하는 일이었습니다. FederationWrapper 패턴을 두어 각 리모트 앱이 React Query, Jotai, Modal 같은 Provider 트리를 자체 보유하도록 했고, 호스트 런타임과 상태를 격리했습니다.
- 구현 과정에서 remoteEntry 캐싱으로 구버전 리모트가 로드되는 문제가 있었고, remoteEntry만 별도로 no-cache로 배포하고 CloudFront invalidation을 CI/CD에 연결해 해결했습니다.

**무엇이 달라졌나요?**

- Vue 3 host 위에 React remote app 2개를 실제로 붙이며, 전면 재작성 없이도 React 전환을 시작할 수 있는 현실적인 구조를 증명했습니다.
- 독립 배포가 가능해졌고, 팀은 안정성을 유지한 채 점진적인 React 전환을 시도할 수 있게 됐습니다.
- 이 프로젝트의 핵심은 기술 선택 자체보다, 조직이 감당 가능한 전환 전략을 프론트엔드 구조로 만들었다는 점에 있었습니다.

### Vue 3 -> React 점진 전환

**왜 이 프로젝트가 필요했나요?**

- Vue 3 기반 프로젝트는 React 대비 채용과 온보딩 난이도가 높고, 생태계 확장성과 도구 선택지가 제한적이라고 판단했습니다.
- 그렇다고 전면 리라이트는 비용과 위험이 너무 컸기 때문에, 서비스 안에서 점진적으로 옮길 수 있는 경로가 필요했습니다.

**저는 이렇게 풀었습니다.**

- 페이지 단위 공존 전략을 제안하고, 시각화 페이지를 React로 분리하는 방식으로 점진 마이그레이션을 주도했습니다.
- 기존 Vue 공통 모듈을 React에서 그대로 사용할 수 없는 문제는 React 버전 공통 모듈을 별도로 구축하고 전환 가이드를 문서화하는 방식으로 풀었습니다.

**무엇이 달라졌나요?**

- 기술 도입이 선언으로 끝나지 않고, 팀이 실제로 옮겨갈 수 있는 기준과 경로가 생겼습니다.
- 이 작업은 프론트엔드 마이그레이션을 코드 문제가 아니라 조직 적응 문제까지 포함한 실행 설계로 바라보고 풀어본 경험이었습니다.

### 사내 지식 하네스

**왜 이 프로젝트가 필요했나요?**

- 개인마다 좋은 방법과 노하우가 생겨도 그것이 팀 전체의 공통 자산으로 남지 않는 경우가 많았습니다.
- 특히 AI 도구와 워크플로우는 사람마다 쓰는 방식의 편차가 컸고, 그 차이가 생산성 편차로 이어지기 쉬웠습니다.

**저는 이렇게 풀었습니다.**

- 팀별·역할별 공통 지식을 축적하고 공유할 수 있는 내부 개발 하네스를 고민했습니다.
- 핵심은 좋은 방법이 개인 경험으로 끝나지 않고, 자유로운 제안과 검토를 거쳐 팀 자산으로 누적되는 구조를 만드는 것이었습니다.
- 외부 공개용이라 세부 구조를 자세히 적기는 어렵지만, 지식이 계속 쌓이고 재사용되는 운영 흐름을 설계하고 실험했습니다.

**무엇이 달라졌나요?**

- 좋은 방법이 개인 노하우로만 남지 않고, 팀이 함께 활용할 수 있는 자산으로 축적되는 기반을 만들 수 있었습니다.
- 이 경험은 단순한 내부 도구 개발보다도, 팀이 어떻게 함께 성장할 수 있는 구조를 만들 것인가에 대한 고민에 더 가까웠습니다.

### 전환 실행 / 개발 생산성

새로운 도구와 방법론은 한 번의 소개로 끝나지 않고, 실험과 검증, 공유의 흐름이 있어야 팀에 실제로 정착한다고 생각합니다. 그래서 React 전환 방향을 제안하는 데서 그치지 않고 직접 실험·구현해 전환 경로를 증명했고, AI 도구와 워크플로우도 실무에서 검증해 팀에 공유했습니다. 그 결과 사내 지식 하네스와 E2E 자동 생성 도구까지 제작하게 되었고, 이 경험은 제가 단순히 기능을 구현하는 사람보다 팀의 실행 방식을 함께 설계하는 사람에 가깝다는 점을 보여줬습니다.

### 아키텍처 시각화 개발 (FE)

**왜 이 프로젝트가 필요했나요?**

- 클라우드 리소스는 서비스별로 흩어져 있고 관계도 복잡해서, 전체 구조를 한눈에 파악하기 어려웠습니다.
- 사용자가 어떤 리소스가 어디와 연결돼 있는지, 무엇이 불필요한지 빠르게 이해할 수 있도록 만드는 시각화가 필요했습니다.
- 리소스를 한눈에 조회할 수 있다면 불필요한 리소스를 찾거나 아키텍처 구조를 판단하는 비용을 크게 줄일 수 있다고 봤습니다.

**저는 이렇게 풀었습니다.**

- 먼저 복잡한 클라우드 데이터를 시각화에 맞는 그래프 구조로 바꾸는 데이터 컨버팅 작업을 설계했습니다.
- 기획 단계에서 어떤 수준까지 구조를 보여줘야 사용자가 실제 판단에 도움을 받을 수 있는지부터 정리했습니다.
- 대규모 리소스 렌더링과 줌/드래그/호버 인터랙션에서의 성능 저하를 해결하기 위해 임계치 기반 축약/클러스터링을 적용했습니다.
- interaction layer와 visualization layer를 분리해 상호 영향도를 낮췄고, 레이아웃/배치 규칙을 직접 조정해 구조 파악이 더 쉬운 방향으로 화면을 개선했습니다.

**무엇이 달라졌나요?**

- 사용자는 전체 구조를 더 빠르게 파악할 수 있었고, 불필요한 리소스와 연결 관계를 확인하는 흐름도 더 쉬워졌습니다.
- 성능 역시 실제 사용 가능한 수준으로 끌어올릴 수 있었습니다.
- 이 프로젝트는 복잡한 데이터를 단순히 그리는 일이 아니라, 사용자가 이해할 수 있는 구조로 번역하는 프론트엔드 설계 문제라는 점을 깊게 체감한 경험이었습니다.

### SSR 개발 경험

**왜 이 프로젝트가 필요했나요?**

- 초기 로딩 시 스타일이 깜빡이는 FOUC 문제와 i18n 처리로 인해 원문이 먼저 노출되는 문제가 있었습니다.

**저는 이렇게 풀었습니다.**

- 서버 사이드에서 데이터를 미리 처리하고, 다국어 처리 역시 서버에서 수행하는 구조를 적용했습니다.

**무엇이 달라졌나요?**

- 초기 화면 품질을 개선했고, 초기 진입 시 느껴지는 불안정함을 줄일 수 있었습니다.

### 대시보드 시각화 라이브러리 제작

**왜 이 프로젝트가 필요했나요?**

- 클라우드 리소스 정보를 한 화면에서 함축적으로 보여줘야 사용자가 빠르게 상황을 파악할 수 있었습니다.
- 화면마다 차트를 따로 만드는 방식으로는 이후 확장에 대응하기 어렵다고 판단했습니다.

**저는 이렇게 풀었습니다.**

- 필요한 데이터와 시각화 방식을 먼저 다시 정리했고, 화면별 정보 우선순위를 재구성했습니다.
- 이후 시각화 전용 디자인 시스템과 컴포넌트 시스템을 함께 설계해 재사용 가능한 구조를 만들었습니다.

**무엇이 달라졌나요?**

- 단순히 차트를 몇 개 추가한 것이 아니라, 이후 대시보드 확장에도 대응할 수 있는 기반을 만들 수 있었습니다.
- 이 경험은 시각화도 결국 시스템으로 다뤄야 한다는 기준을 세우게 해줬습니다.

### 디자인 시스템 구축 작업

**왜 이 프로젝트가 필요했나요?**

- 페이지와 기능이 늘어나면서 동일한 컴포넌트가 중복 구현되고, 같은 역할을 하는 UI가 다르게 제공되는 문제가 있었습니다.
- 기존 디자인 시스템도 존재했지만 한계가 분명해 재개편이 필요한 상태였습니다.

**저는 이렇게 풀었습니다.**

- 먼저 디자인팀과 공통된 비전과 원칙을 맞추는 데 시간을 썼습니다.
- 이후 좋은 컴포넌트 디자인 패턴을 조사하고, 시스템을 다시 구성하는 방향으로 작업을 진행했습니다.

**무엇이 달라졌나요?**

- 이 작업은 단순한 UI 정리를 넘어, 팀 간 기준을 맞추고 확장 가능한 공통 규칙을 정리하는 경험이 됐습니다.

### 문서 출력 개발 (BE)

**왜 이 프로젝트가 필요했나요?**

- 화면에 보이는 데이터를 문서 형태로 남겨 외부 공유나 증적 관리에 활용할 수 있어야 했습니다.
- 단순한 파일 출력이 아니라, 어떤 서버가 어떤 책임을 가져야 하는지부터 정리할 필요가 있었습니다.

**저는 이렇게 풀었습니다.**

- 요구사항을 기준으로 기능 책임을 먼저 나누고, 포맷별 문서 출력 기능과 export 플로우를 설계·구현했습니다.
- 구현 과정에서는 출력 시간이 과하게 길어지는 병목과 메모리 누수 문제를 함께 해결했습니다.

**무엇이 달라졌나요?**

- 프론트엔드 관점의 화면 기능을 백엔드 출력 플로우까지 확장해 설계해보는 경험을 할 수 있었습니다.

## 오픈소스 / 개인 프로젝트

### Tenetx

**왜 이 프로젝트가 필요했나요?**

- AI 코딩 도구 발전으로 워크플로우, 에이전트, 스킬 활용이 점점 보편화되면서 개발자 간 구현 격차가 줄어들고 있다고 느꼈습니다.
- 그 안에서 앞으로의 차별점은 구현 속도보다, 개발 철학과 방법론을 코드로 선언하고 재사용 가능한 시스템으로 만드는 데 있다고 봤습니다.

**저는 이렇게 개발했습니다.**

- 엔지니어링 원칙을 `philosophy.yaml`로 선언하면 hooks, 모델 라우팅, 에이전트 파이프라인이 생성되는 philosophy-driven 하네스를 설계했습니다.
- 프롬프트의 어휘, 구조, 컨텍스트에서 16개 신호를 추출해 Haiku, Sonnet, Opus를 자동 라우팅하는 비용 최적화 엔진을 구현했습니다.
- 세션별 패턴과 솔루션을 축적하는 Compound Engineering Loop를 설계해, 사용 패턴이 쌓일수록 하네스가 더 좋아지는 구조를 만들었습니다.

**프로젝트 규모**

- 19개 에이전트 문서, 21개 스킬, 9개 실행 모드, 8개 MCP 서버
- 테스트 1,204개, npm 배포, MIT 라이선스 기반 운영

**이 프로젝트가 저에게 중요한 이유**

- Tenetx는 단순한 사이드 프로젝트가 아니라, 공통화와 지식 축적, 워크플로우 설계에 대한 관심이 오픈소스 수준의 구조로 이어진 사례입니다.

### PICode

**왜 이 프로젝트가 필요했나요?**

- 프로젝트를 진행할 때 Slack, Git, Notion 등 여러 도구를 함께 써야 하면서 정보가 산재되고 맥락이 끊기는 문제가 있었습니다.
- "이 기능들을 한 서비스 안에서 제공하면 더 편하게 협업할 수 있지 않을까"라는 질문에서 출발했습니다.

**저는 이렇게 개발했습니다.**

- 문서 편집, Docker 시각화, 코드 동시 편집 기능을 포함한 협업 플랫폼을 구현했습니다.
- 개발 과정에서는 동시 편집 시 사용자 간 코드 충돌 문제도 겪었고, 이를 해결하며 협업 기능의 난도를 체감했습니다.

**무엇이 달라졌나요?**

- 공개 SW 개발자 대회 일반부문 금상을 수상했습니다.
- 초기부터 협업과 시각화, 통합 경험에 관심이 있었다는 점을 보여주는 프로젝트라고 생각합니다.

### CRMS

**왜 이 프로젝트가 필요했나요?**

- 클라우드 리소스를 한 번에 확인하고 관리하기 어려운 문제를 해결하고 싶었습니다.
- 여러 서비스와 페이지를 오가며 수동으로 점검해야 하는 불편함이 컸고, 이를 더 편하게 만들 수 있다고 생각했습니다.

**저는 이렇게 개발했습니다.**

- 핵심 리소스를 한 화면에서 확인할 수 있는 Dashboard, 리소스 관계를 빠르게 파악할 수 있는 아키텍처 시각화, 다양한 리소스 타입을 탐색·정렬·필터링할 수 있는 Data Grid를 구현했습니다.
- 데이터 유형이 너무 다양해 표현 방식이 복잡해지는 문제는 공통 데이터 모델과 표현 규칙을 먼저 정리한 뒤 컴포넌트를 확장 가능한 구조로 설계해 풀었습니다.

**무엇이 달라졌나요?**

- 공개 SW 개발자 대회 일반부문 은상을 수상했습니다.
- 이후 실무에서 계속 이어진 시각화와 구조 파악 문제 해결의 출발점이 된 경험입니다.
